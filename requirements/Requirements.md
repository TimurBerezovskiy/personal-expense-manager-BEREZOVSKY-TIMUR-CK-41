# Вимоги до вебзастосунку  
# “Менеджер особистих витрат”

## 1. Призначення системи

Вебзастосунок “Менеджер особистих витрат” призначений для обліку, збереження, перегляду та контролю особистих витрат користувача.

Система має допомагати користувачу:

створювати записи про витрати;  
задавати для них суму, категорію, дату та опис;  
переглядати список витрат;  
фільтрувати витрати за категорією та періодом;  
бачити загальну суму витрат;  
контролювати перевищення встановлених лімітів за категоріями.

## 2. Межі системи

Система є однокористувацьким вебзастосунком, що працює в браузері без обов’язкового серверного бекенду.

До обсягу системи входить:

створення витрат;  
редагування витрат;  
видалення витрат;  
перегляд списку витрат;  
фільтрація витрат за категорією;  
фільтрація витрат за періодом;  
розрахунок загальної суми витрат;  
розрахунок сум за категоріями;  
встановлення та редагування лімітів за категоріями;  
відображення попереджень про перевищення лімітів;  
збереження даних локально в браузері.

Поза обсягом системи:

багатокористувацька робота;  
реєстрація та авторизація користувачів;  
синхронізація між пристроями через хмару;  
інтеграція з банківськими сервісами;  
автоматичне отримання транзакцій з банківської картки;  
обробка реальних платежів;  
збереження даних на сервері;  
експорт даних у PDF, Excel або Google Sheets.

## 3. Роль користувача

Користувач:  
Людина, яка веде облік власних особистих витрат і хоче контролювати свої витрати за категоріями та лімітами.

## 5. Функціональні вимоги

### FR-01. Створення витрати

Система повинна надавати користувачу форму для створення нового запису про витрату.

Під час створення витрати користувач повинен мати можливість вказати:

назву витрати;  
суму витрати;  
категорію;  
дату витрати;  
опис.

Після успішного створення:

витрата додається до списку витрат;  
їй автоматично присвоюється унікальний ID;  
фіксується дата створення;  
оновлюється загальна сума витрат;  
оновлюється сума витрат у відповідній категорії.

### FR-02. Валідація форми створення та редагування витрати

Система повинна перевіряти коректність введених даних перед збереженням витрати.

У разі невалідного вводу система повинна:

не зберігати витрату;  
показувати повідомлення про помилку біля відповідного поля або у верхній частині форми.

### FR-03. Перегляд списку витрат

Система повинна відображати список усіх створених витрат.

Для кожної витрати в списку мають відображатися:

назва;  
сума;  
категорія;  
дата витрати;  
опис.

### FR-04. Редагування витрати

Система повинна дозволяти користувачу редагувати наявну витрату.

Користувач повинен мати можливість змінювати:

назву;  
суму;  
категорію;  
дату витрати;  
опис.

Після успішного редагування система повинна:

зберегти оновлені дані;  
оновити дату останнього редагування;  
відобразити змінені значення у списку витрат;  
перерахувати загальну суму витрат;  
перерахувати суми витрат за категоріями.

### FR-05. Видалення витрати

Система повинна дозволяти користувачу видалити витрату зі списку.

Перед видаленням система повинна запитати підтвердження дії.

Після підтвердження:

витрата видаляється зі списку;  
витрата видаляється з локального сховища;  
загальна сума витрат перераховується;  
сума витрат у відповідній категорії перераховується.

### FR-06. Фільтрація витрат за категорією

Система повинна дозволяти користувачу фільтрувати список витрат за категорією.

Користувач повинен мати можливість:

обрати одну категорію;  
обрати варіант “All categories” для перегляду всіх витрат.

Після застосування фільтра система повинна відображати тільки ті витрати, які відповідають обраній категорії.

### FR-07. Фільтрація витрат за періодом

Система повинна дозволяти користувачу фільтрувати список витрат за періодом.

Користувач повинен мати можливість вказати:

початкову дату;  
кінцеву дату.

Після застосування фільтра система повинна відображати тільки ті витрати, дата яких входить у вибраний період.

### FR-08. Комбінована фільтрація витрат

Система повинна дозволяти одночасно застосовувати фільтр за категорією та фільтр за періодом.

Якщо активні обидва фільтри, у списку мають відображатися тільки витрати, які одночасно відповідають вибраній категорії та вибраному періоду.

### FR-09. Відображення загальної суми витрат

Система повинна відображати загальну суму витрат.

Загальна сума повинна оновлюватися після:

створення витрати;  
редагування витрати;  
видалення витрати;  
застосування фільтрів;  
очищення фільтрів.

Якщо активні фільтри, загальна сума повинна розраховуватися тільки для відфільтрованих витрат.

### FR-10. Відображення сум за категоріями

Система повинна відображати суму витрат окремо для кожної категорії.

Для кожної категорії система повинна показувати:

назву категорії;  
загальну суму витрат у цій категорії;  
інформацію про ліміт, якщо він встановлений.

### FR-11. Встановлення ліміту для категорії

Система повинна дозволяти користувачу встановлювати ліміт витрат для категорії.

Під час створення або редагування ліміту користувач повинен мати можливість вказати:

категорію;  
суму ліміту.

Після збереження ліміту система повинна використовувати його для контролю витрат у відповідній категорії.

### FR-12. Контроль перевищення ліміту

Система повинна автоматично перевіряти, чи перевищує сума витрат у категорії встановлений ліміт.

Якщо сума витрат у категорії перевищує ліміт, система повинна показати попередження.

Попередження має містити:

назву категорії;  
встановлений ліміт;  
фактичну суму витрат;  
суму перевищення.

### FR-13. Збереження даних між сесіями

Система повинна зберігати витрати, категорії та ліміти у локальному сховищі браузера.

Після повторного відкриття застосунку раніше збережені дані мають бути відновлені.

### FR-14. Очищення фільтрів

Система повинна надавати користувачу можливість одним натисканням скинути всі активні фільтри та повернутися до повного списку витрат.

### FR-15. Скидання даних

Система повинна надавати користувачу можливість скинути всі збережені витрати та ліміти.

Перед скиданням система повинна запитати підтвердження дії.

Після підтвердження:

усі користувацькі витрати видаляються;  
ліміти видаляються або повертаються до демонстраційних значень;  
список витрат оновлюється;  
підсумкові суми перераховуються.

## 6. Бізнес-правила

### BR-01. Обов’язкові поля

Під час створення витрати обов’язковими є такі поля:

назва;  
сума;  
категорія;  
дата витрати.

### BR-02. Назва витрати

Назва витрати:

повинна містити від 3 до 100 символів;  
не повинна складатися лише з пробілів.

### BR-03. Опис витрати

Опис витрати:

є необов’язковим;  
якщо заповнений, повинен містити не більше 300 символів.

### BR-04. Сума витрати

Сума витрати:

не може бути порожньою;  
повинна бути числом;  
повинна бути більшою за 0;  
не може перевищувати 1 000 000 грн.

### BR-05. Категорія витрати

Під час створення витрати категорія:

не може бути порожньою;  
повинна бути вибрана зі списку доступних категорій.

### BR-06. Дата витрати

Під час створення витрати дата:

не може бути порожньою;  
не може бути пізнішою за поточну дату.

### BR-07. Доступні категорії

Для витрати допускаються лише такі категорії:

Food  
Transport  
Education  
Health  
Entertainment  
Shopping  
Other

### BR-08. Ліміт категорії

Ліміт категорії:

є необов’язковим;  
якщо вказаний, повинен бути числом;  
повинен бути більшим за 0;  
не може перевищувати 10 000 000 грн.

### BR-09. Перевищення ліміту

Ліміт вважається перевищеним, якщо сума витрат у категорії більша за встановлений ліміт.

Якщо сума витрат дорівнює ліміту, ліміт не вважається перевищеним.

### BR-10. Розрахунок загальної суми

Загальна сума витрат повинна дорівнювати сумі всіх витрат, які відповідають активним фільтрам.

Якщо фільтри не застосовані, загальна сума повинна враховувати всі збережені витрати.

### BR-11. Розрахунок суми за категорією

Сума витрат за категорією повинна дорівнювати сумі всіх витрат, що належать до цієї категорії.

Якщо активний фільтр за періодом, сума за категорією повинна враховувати тільки витрати в межах цього періоду.

### BR-12. Видалення витрати

Будь-яка витрата може бути видалена тільки після явного підтвердження користувача.

### BR-13. Порожній список витрат

Якщо у системі немає витрат або жодна витрата не відповідає активним фільтрам, система повинна показувати повідомлення про відсутність витрат.

### BR-14. Формат відображення суми

Усі суми в інтерфейсі повинні відображатися у гривнях з двома знаками після коми.

Приклад:

125.50 грн

## 7. Правила відображення станів

### SR-01.

Якщо сума витрат у категорії перевищує встановлений ліміт, система повинна:

візуально позначити категорію як таку, що перевищила ліміт;  
показати попередження користувачу;  
відобразити суму перевищення;  
врахувати перевищення у блоці підсумків.

### SR-02.

Категорії з перевищеним лімітом повинні бути візуально помітнішими за інші.

### SR-03.

Якщо список витрат порожній, система повинна показати повідомлення:

“Немає витрат для відображення”

### SR-04.

Якщо користувач застосував фільтри, система повинна візуально показати, що список витрат відображається не повністю, а відповідно до активних фільтрів.

## 8. Нефункціональні вимоги

### NFR-01. Тип застосунку

Система повинна бути реалізована як вебзастосунок, доступний через браузер.

### NFR-02. Сумісність

Система повинна коректно працювати в актуальних версіях:

Google Chrome  
Mozilla Firefox  
Microsoft Edge  
Safari

### NFR-03. Адаптивність

Інтерфейс повинен бути придатним для використання щонайменше на таких розмірах екрана:

1920×1080  
1366×768  
390×844

### NFR-04. Продуктивність

Після відкриття застосунку список витрат повинен ставати доступним для взаємодії не довше ніж за 2 секунди за наявності до 300 витрат у локальному сховищі.

### NFR-05. Збереження даних

Після перезавантаження сторінки збережені витрати та ліміти не повинні втрачатися, якщо користувач не видалив їх явно.

### NFR-06. Зрозумілість повідомлень

Усі повідомлення про помилки валідації повинні бути сформульовані зрозуміло і прямо вказувати, що саме введено неправильно.

### NFR-07. Надійність

У разі відсутності даних у локальному сховищі система повинна коректно відкриватися і показувати порожній список без критичної помилки.

### NFR-08. Сумісність з GitHub Pages

Система повинна працювати без серверної частини та бути придатною для публікації на GitHub Pages.

### NFR-09. Безпека даних

Система не повинна вимагати введення паролів, банківських карток, номерів рахунків або інших чутливих фінансових даних.

## 10. Критерії приймання для ключових функцій

### AC-01. Створення валідної витрати

Передумова: користувач відкрив форму створення витрати.  
Умова: користувач заповнив обов’язкові поля коректними даними.  
Очікуваний результат: після натискання кнопки збереження нова витрата додається до списку витрат, а загальна сума оновлюється.

### AC-02. Заборона створення невалідної витрати

Передумова: користувач відкрив форму створення витрати.  
Умова: хоча б одне обов’язкове поле порожнє або містить невалідне значення.  
Очікуваний результат: витрата не створюється, система показує повідомлення про помилку.

### AC-03. Редагування витрати

Передумова: у системі існує хоча б одна витрата.  
Умова: користувач змінює дані витрати та зберігає зміни.  
Очікуваний результат: витрата оновлюється у списку, дата останнього редагування змінюється, підсумкові суми перераховуються.

### AC-04. Видалення витрати

Передумова: у системі існує хоча б одна витрата.  
Умова: користувач натискає кнопку видалення та підтверджує дію.  
Очікуваний результат: витрата видаляється зі списку та локального сховища, загальна сума витрат перераховується.

### AC-05. Фільтрація за категорією

Передумова: у списку існують витрати з різними категоріями.  
Умова: користувач застосовує фільтр, наприклад “Food”.  
Очікуваний результат: у списку залишаються тільки витрати, що належать до категорії “Food”.

### AC-06. Фільтрація за періодом

Передумова: у списку існують витрати з різними датами.  
Умова: користувач задає початкову та кінцеву дату періоду.  
Очікуваний результат: у списку залишаються тільки витрати, дата яких входить у вибраний період.

### AC-07. Розрахунок загальної суми

Передумова: у системі існує одна або більше витрат.  
Умова: користувач створює, редагує, видаляє витрату або застосовує фільтри.  
Очікуваний результат: загальна сума витрат автоматично перераховується відповідно до актуального списку витрат.

### AC-08. Контроль перевищення ліміту

Передумова: для категорії встановлено ліміт.  
Умова: сума витрат у цій категорії перевищує встановлений ліміт.  
Очікуваний результат: система показує попередження про перевищення ліміту та відображає суму перевищення.

### AC-09. Збереження між сесіями

Передумова: користувач створив одну або більше витрат.  
Умова: користувач перезавантажує сторінку або повторно відкриває застосунок.  
Очікуваний результат: раніше створені витрати залишаються в системі.

### AC-10. Очищення фільтрів

Передумова: користувач застосував один або більше фільтрів.  
Умова: користувач натискає кнопку очищення фільтрів.  
Очікуваний результат: усі активні фільтри скидаються, а у списку знову відображаються всі витрати.

## 11. Припущення та обмеження

Система не використовує реальні банківські або платіжні дані.  
Система не потребує серверної частини для базової роботи.  
Усі дані зберігаються локально в браузері поточного користувача.  
Якщо користувач очистить local storage браузера, дані можуть бути втрачені.  
Система розрахована на навчальний сценарій використання, а не на промислову фінансову систему.  
Система не виконує реальні платежі.  
Система не підключається до банківських API.  
Система не підтримує синхронізацію між пристроями.  
Система повинна бути придатною для розгортання на GitHub Pages.
